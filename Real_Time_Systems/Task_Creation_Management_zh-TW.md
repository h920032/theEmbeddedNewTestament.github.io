## **概念 → 為何重要 → 最小範例 → 動手試試 → 重點摘要**

**概念**：任務就像工廠中的工人，每個人都有特定的工作要完成。RTOS 就像一位聰明的管理者，確保每位工人在時限內獲得適當的時間與資源來完成工作。

**為何重要**：如果沒有適當的任務管理，你的嵌入式系統就像一間沒有管理者的工廠——工人互相碰撞、資源被浪費、截止時間被錯過。良好的任務管理可以確保一切順暢且可預測地運行。

**最小範例**：一個簡單的系統包含三個任務：一個每 100ms 閃爍 LED，另一個每 500ms 讀取感測器，第三個每 1000ms 發送資料。每個任務獨立運行，但系統協調它們全部。

**動手試試**：從單一任務開始，然後新增更多任務並觀察 RTOS 如何管理它們。改變優先權，看看它如何影響執行順序。

**重點摘要**：任務管理的核心在於設計能有效協作的獨立工作者，由 RTOS 處理複雜的協調工作，讓你可以專注於每個任務應該做什麼。

## 📋 **快速參考：關鍵要點**

### **任務基礎**
- **獨立執行**：每個任務以自己的上下文獨立執行
- **並行運作**：多個任務可以同時處於就緒狀態
- **排程執行**：RTOS 排程器決定何時執行哪個任務
- **資源共享**：任務可以共享系統資源並進行通訊
- **基於優先權**：較高優先權的任務可以中斷較低優先權的任務

### **任務狀態**
- **已建立（Created）**：任務存在但尚未被排程
- **就緒（Ready）**：任務已準備好執行，等待 CPU 時間
- **執行中（Running）**：任務正在 CPU 上執行
- **阻塞（Blocked）**：任務正在等待某件事（延遲、資料、資源）
- **已刪除（Deleted）**：任務已從系統中移除

### **任務設計原則**
- **單一職責**：每個任務應有一個明確的工作
- **適當粒度**：不要太大，也不要太小
- **獨立運作**：任務應能獨立運行
- **清晰的通訊**：任務之間有定義良好的介面
- **資源效率**：適當的堆疊大小與記憶體使用

### **優先權指引**
- **高優先權（4-5）**：緊急功能、安全關鍵操作
- **中優先權（2-3）**：正常系統操作、資料處理
- **低優先權（1）**：背景任務、狀態更新、清理工作
- **優先權繼承**：任務可從其存取的資源繼承優先權
- **優先權反轉**：低優先權任務可能阻塞高優先權任務

## 🧠 **核心概念**

### **什麼是任務？**

任務是一個與其他任務並行執行的獨立工作單元。可以把它想像成一個能與其他程式同時運行的獨立程式。

```
┌─────────────────────────────────────────────────────────────┐
│                     什麼是任務？                            │
├─────────────────────────────────────────────────────────────┤
│                                                           │
│  傳統程式：                                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  main() {                                          │   │
│  │    while(1) {                                      │   │
│  │      blink_led();                                  │   │
│  │      read_sensor();                                │   │
│  │      send_data();                                  │   │
│  │      delay(100);                                   │   │
│  │    }                                               │   │
│  │  }                                                 │   │
│  └─────────────────────────────────────────────────────┘   │
│  ❌ 所有操作依序執行                                      │
│  ❌ 無法為不同操作設定優先權                               │
│  ❌ 難以讓操作彼此獨立                                    │
│                                                           │
│  使用 RTOS 的任務：                                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│  │ 任務 1：    │  │ 任務 2：    │  │ 任務 3：    │        │
│  │ 閃爍 LED    │  │ 讀取感測器  │  │ 發送資料    │        │
│  │ 每 100ms    │  │ 每 500ms    │  │ 每 1000ms   │        │
│  └─────────────┘  └─────────────┘  └─────────────┘        │
│  ✅ 每個任務獨立運行                                      │
│  ✅ 可設定不同的優先權                                    │
│  ✅ 容易新增/移除/修改任務                                │
└─────────────────────────────────────────────────────────────┘
```

### **為什麼使用任務而非單一大迴圈？**

**單一迴圈的問題：**
```
┌─────────────────────────────────────────────────────────────┐
│                    單一迴圈的問題                           │
├─────────────────────────────────────────────────────────────┤
│                                                           │
│  while(1) {                                               │
│    // 如果感測器讀取耗時過長怎麼辦？                       │
│    sensor_value = read_sensor();                          │
│                                                           │
│    // LED 閃爍會被延遲！                                  │
│    blink_led();                                           │
│                                                           │
│    // 如果資料發送失敗怎麼辦？                             │
│    send_data();                                           │
│                                                           │
│    // 所有操作都在互相等待                                 │
│    delay(100);                                            │
│  }                                                        │
│                                                           │
│  ❌ 一個慢速操作會延遲所有事情                             │
│  ❌ 無法為關鍵操作設定優先權                               │
│  ❌ 難以優雅地處理故障                                    │
│  ❌ 時序變得不可預測                                      │
└─────────────────────────────────────────────────────────────┘
```

**基於任務的優點：**
```
┌─────────────────────────────────────────────────────────────┐
│                    基於任務的優點                           │
├─────────────────────────────────────────────────────────────┤
│                                                           │
│  任務 1（高優先權）：                                     │
│  while(1) {                                               │
│    blink_led();                                           │
│    vTaskDelay(100);                                       │
│  }                                                        │
│                                                           │
│  任務 2（中優先權）：                                     │
│  while(1) {                                               │
│    sensor_value = read_sensor();                          │
│    vTaskDelay(500);                                       │
│  }                                                        │
│                                                           │
│  任務 3（低優先權）：                                     │
│  while(1) {                                               │
│    send_data();                                           │
│    vTaskDelay(1000);                                      │
│  }                                                        │
│                                                           │
│  ✅ 每個任務按自己的節奏運行                               │
│  ✅ 關鍵操作（LED）不會被延遲                              │
│  ✅ 一個任務的故障不會影響其他任務                          │
│  ✅ 時序可預測且可靠                                      │
└─────────────────────────────────────────────────────────────┘
```

## 🔄 **任務生命週期**

### **任務狀態**

任務在其生命週期中會經歷不同的狀態：

```
┌─────────────────────────────────────────────────────────────┐
│                    任務狀態機                               │
├─────────────────────────────────────────────────────────────┤
│                                                           │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                │
│  │ 已建立  │───▶│  就緒   │───▶│ 執行中  │                │
│  └─────────┘    └─────────┘    └────┬────┘                │
│                                      │                     │
│                                      ▼                     │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                │
│  │ 已刪除  │◄───│  阻塞   │◄───│         │                │
│  └─────────┘    └─────────┘    └─────────┘                │
│                                                           │
│  狀態轉換：                                                │
│  • 已建立 → 就緒：任務準備好可以執行                       │
│  • 就緒 → 執行中：排程器選擇此任務                         │
│  • 執行中 → 就緒：任務讓出 CPU 或被搶佔                    │
│  • 執行中 → 阻塞：任務等待某件事                           │
│  • 阻塞 → 就緒：任務等待的事件發生了                       │
│  • 任意狀態 → 已刪除：任務從系統中移除                      │
└─────────────────────────────────────────────────────────────┘
```

### **每個狀態中發生什麼事？**

**已建立（Created）**：任務存在但尚未被排程
**就緒（Ready）**：任務已準備好執行，等待 CPU 時間
**執行中（Running）**：任務正在 CPU 上執行
**阻塞（Blocked）**：任務正在等待某件事（延遲、資料、資源）
**已刪除（Deleted）**：任務已被移除，不會再執行

## 🎨 **任務設計原則**

### **單一職責原則**

每個任務應該有一個明確的工作：

```
┌─────────────────────────────────────────────────────────────┐
│                    好的設計 vs 壞的設計                     │
├─────────────────────────────────────────────────────────────┤
│                                                           │
│  ❌ 壞的設計：一個任務做所有事情                           │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ void vDoEverythingTask(void *pvParameters) {       │   │
│  │   while(1) {                                        │   │
│  │     // 太多職責了！                                  │   │
│  │     read_sensors();                                  │   │
│  │     process_data();                                  │   │
│  │     control_motors();                                │   │
│  │     update_display();                                │   │
│  │     send_telemetry();                                │   │
│  │     vTaskDelay(100);                                 │   │
│  │   }                                                  │   │
│  │ }                                                    │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                           │
│  ✅ 好的設計：每個任務有一個工作                           │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│  │ 感測器任務  │  │ 控制        │  │ 顯示        │        │
│  │ 每 100ms    │  │ 任務        │  │ 任務        │        │
│  │ 讀取資料    │  │ 處理與      │  │ 每 50ms     │        │
│  └─────────────┘  │ 控制        │  │ 更新 UI     │        │
│                   │ 每 50ms     │  └─────────────┘        │
│                   └─────────────┘                         │
│                                                           │
│  好的設計的優點：                                          │
│  • 更容易理解和除錯                                       │
│  • 可以獨立優化每個任務                                    │
│  • 更容易測試個別元件                                      │
│  • 更靈活且易於維護                                       │
└─────────────────────────────────────────────────────────────┘
```

### **任務粒度**

任務不應該太大也不應該太小：

**太大（粗粒度）：**
- 難以理解和除錯
- 難以優化
- 回應性差
- 難以測試

**太小（細粒度）：**
- 過多的上下文切換
- 記憶體額外開銷
- 複雜的協調
- 難以管理

**剛剛好：**
- 清晰、專注的職責
- 適當的執行頻率
- 可管理的複雜度
- 良好的可測試性

## 🎯 **優先權管理**

### **為什麼優先權重要**

當多個任務都處於就緒狀態時，優先權決定哪個任務先執行：

```
┌─────────────────────────────────────────────────────────────┐
│                      優先權系統                            │
├─────────────────────────────────────────────────────────────┤
│                                                           │
│  優先權 5：緊急停止（最高）                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ ██████████████████████████████████████████████████ │   │
│  │ 緊急停止 - 必須立即執行！                           │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                           │
│  優先權 4：安全監控                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ ██████████████████████████████████████████████████ │   │
│  │ 安全檢查 - 對系統安全至關重要                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                           │
│  優先權 3：控制迴圈                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ ██████████████████████████████████████████████████ │   │
│  │ 主要控制 - 維持系統運行                             │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                           │
│  優先權 2：資料記錄                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ ██████████████████████████████████████████████████ │   │
│  │ 記錄 - 重要但非關鍵                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                           │
│  優先權 1：狀態更新（最低）                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ ██████████████████████████████████████████████████ │   │
│  │ 狀態 - 有的話很好但非必要                           │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                           │
│  ⚠️  較高優先權的任務可以中斷較低優先權的任務！            │
└─────────────────────────────────────────────────────────────┘
```

### **優先權分配指引**

**高優先權（4-5）：**
- 緊急功能
- 安全關鍵操作
- 即時控制迴圈
- 中斷處理器

**中優先權（2-3）：**
- 正常系統操作
- 資料處理
- 通訊任務
- 常規監控

**低優先權（1）：**
- 背景任務
- 狀態更新
- 非關鍵性記錄
- 清理操作

## 💾 **資源管理**

### **堆疊管理**

每個任務都需要自己的堆疊空間：

```
┌─────────────────────────────────────────────────────────────┐
│                    任務堆疊配置                             │
├─────────────────────────────────────────────────────────────┤
│                                                           │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                系統記憶體                            │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐ │   │
│  │  │ 任務 1      │  │ 任務 2      │  │ 任務 3      │ │   │
│  │  │ 堆疊        │  │ 堆疊        │  │ 堆疊        │ │   │
│  │  │ 1KB         │  │ 2KB         │  │ 512B        │ │   │
│  │  └─────────────┘  └─────────────┘  └─────────────┘ │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │                堆積（共享）                          │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │                全域變數                              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                           │
│  堆疊大小的考量：                                          │
│  • 太小：堆疊溢位導致當機                                 │
│  • 太大：浪費記憶體                                       │
│  • 使用堆疊檢查功能監控實際使用量                          │
│  • 考慮函式呼叫深度和區域變數                              │
└─────────────────────────────────────────────────────────────┘
```

### **記憶體配置策略**

**靜態配置：**
- 在編譯時期配置所有記憶體
- 可預測的記憶體使用量
- 不會有執行時期配置失敗
- 較不靈活

**動態配置：**
- 依需求配置記憶體
- 較為靈活
- 有配置失敗的風險
- 記憶體碎片化

## 🧪 **引導式實驗**

### **實驗 1：單一任務系統**
**目標**：理解基本的任務建立與執行。

**準備**：建立一個執行簡單操作的單一任務。

**步驟**：
1. 建立一個閃爍 LED 的任務函式
2. 以適當的堆疊大小和優先權設定任務
3. 啟動排程器並觀察任務執行
4. 修改任務的時序並觀察變化

**預期結果**：理解基本的任務建立，以及任務如何獨立運行。

### **實驗 2：多任務協調**
**目標**：學習多個任務如何協同工作。

**準備**：建立兩個需要協調工作的任務。

**步驟**：
1. 建立一個產生資料的生產者任務
2. 建立一個處理資料的消費者任務
3. 使用佇列在任務之間傳遞資料
4. 觀察任務如何在不互相干擾的情況下協調

**預期結果**：理解任務的通訊與協調。

### **實驗 3：優先權管理**
**目標**：學習優先權如何影響任務執行。

**準備**：建立具有不同優先權的多個任務。

**步驟**：
1. 建立具有不同優先權的任務
2. 讓高優先權任務頻繁執行
3. 讓低優先權任務執行較繁重的工作
4. 觀察優先權如何影響執行順序

**預期結果**：理解基於優先權的排程及其效果。
