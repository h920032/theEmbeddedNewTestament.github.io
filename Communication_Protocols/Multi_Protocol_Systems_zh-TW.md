# 多協定系統

嵌入式產品經常需要橋接、轉譯或協調多種協定（例如 UART↔CAN、CAN↔Ethernet、I2C↔SPI）。本文件提供設計可靠、可維護的多協定系統的模式，確保在負載下具有可預測的行為。

---

## 🧠 **概念優先**

### **協定轉譯 vs 協定橋接**
**概念**：轉譯是在協定之間進行格式轉換，橋接則是連接協定而不進行轉換。
**重要性**：理解這個區別有助於為多協定系統選擇正確的架構。
**最小範例**：比較 UART 轉 CAN 轉譯器與原生支援兩種協定的系統。
**動手試試**：針對相同的通訊需求設計兩種方法。
**重點整理**：轉譯增加了複雜度但提供靈活性，橋接較簡單但靈活性較低。

### **資源競爭管理**
**概念**：多個協定會競爭共享資源（CPU、記憶體、頻寬）。
**重要性**：如果沒有適當的管理，一個協定可能會使其他協定資源匱乏，導致系統故障。
**最小範例**：設計一個 UART 和 SPI 共享同一顆 CPU 的系統。
**動手試試**：實作資源分配策略並衡量其效果。
**重點整理**：資源管理對於多協定的可預測效能至關重要。

## 設計目標與理念

### 為什麼多協定系統很重要
現代嵌入式系統必須與各種裝置和網路通訊，每種裝置和網路使用不同的協定，這些協定針對其特定用途進行了最佳化。多協定系統能夠實現：

**互通性**
- **舊系統整合**：連接使用舊版協定的現有系統
- **產業標準**：支援多種產業標準協定
- **供應商多樣性**：與不同製造商的裝置進行介面連接
- **前瞻性**：隨著新協定的出現進行調適

**效能最佳化**
- **協定匹配**：為每條通訊路徑使用最佳協定
- **頻寬最佳化**：將協定能力與資料需求相匹配
- **延遲最佳化**：根據時序需求選擇協定
- **功耗最佳化**：根據功耗限制選擇協定

**系統架構靈活性**
- **模組化設計**：無需大幅重新設計即可新增/移除協定支援
- **可擴展性**：隨著系統成長支援額外的協定
- **可維護性**：隔離特定協定的程式碼以便於維護
- **測試**：獨立測試協定及其組合

### 系統設計原則
**關注點分離**
每個協定都應由專用元件處理，這些元件可以獨立開發、測試和維護。

**抽象層**
通用介面應抽象化協定差異，讓上層應用程式可以與任何支援的協定協同工作。

**資源管理**
系統資源（記憶體、CPU、頻寬）必須跨所有協定進行管理，以確保可預測的效能。

**錯誤隔離**
一個協定中的故障不應影響其他協定的運作。

## 參考架構

### 硬體介面考量
**介面選擇準則**
- **效能需求**：頻寬、延遲、可靠性需求
- **功耗限制**：主動式 vs 被動式介面、休眠模式
- **成本考量**：硬體複雜度、授權費用
- **環境因素**：溫度、濕度、EMI 需求

**常見介面類型**
- **序列介面**：UART、SPI、I2C 用於短距離、低速通訊
- **場域匯流排**：CAN、LIN、Profibus 用於工業應用
- **網路介面**：Ethernet、Wi-Fi 用於長距離、高速通訊
- **無線介面**：Bluetooth、Zigbee、LoRa 用於行動應用

**介面整合挑戰**
- **訊號完整性**：多個介面可能相互干擾
- **接地**：適當的接地分離可防止串擾
- **電源供應**：多個介面可能需要不同的電壓位準
- **EMI/EMC**：多個介面會增加電磁干擾

### 軟體架構層級
**驅動層**
- **硬體抽象**：與實體硬體進行介面連接
- **中斷處理**：高效管理硬體中斷
- **DMA 管理**：最佳化資料傳輸效能
- **錯誤處理**：偵測和回報硬體錯誤

**I/O 佇列層**
- **緩衝區管理**：高效的記憶體分配和釋放
- **流量控制**：防止緩衝區溢位和不足
- **優先權處理**：管理不同的優先權等級
- **逾時管理**：處理通訊逾時

**協定配接器層**
- **協定解析**：在協定格式之間轉換
- **資料驗證**：確保資料完整性和格式合規性
- **錯誤修正**：處理特定協定的錯誤狀況
- **狀態管理**：追蹤協定狀態機

**路由層**
- **訊息路由**：將訊息導向適當的目的地
- **協定轉譯**：在不同協定格式之間轉換
- **負載平衡**：將負載分散到多個介面
- **故障轉移處理**：在需要時切換到備用介面

**應用服務層**
- **業務邏輯**：實作特定應用的功能
- **資料處理**：轉換和分析資料
- **組態管理**：處理系統組態
- **監控與記錄**：追蹤系統效能和事件

### RTOS 整合
**任務架構**
- **介面任務**：每個協定介面的專用任務
- **路由任務**：用於訊息路由和協定轉譯的中央任務
- **應用任務**：用於業務邏輯和資料處理的任務
- **系統任務**：用於系統管理和監控的任務

**同步機制**
- **訊息佇列**：任務間的非同步通訊
- **號誌**：資源存取控制
- **互斥鎖**：對共享資源的獨占存取
- **事件群組**：複雜的事件同步

**記憶體管理**
- **記憶體池**：預先分配的記憶體以達到可預測的效能
- **緩衝區管理**：高效的緩衝區分配和釋放
- **記憶體保護**：防止記憶體損壞和存取違規
- **垃圾收集**：自動記憶體清理（如適用）

```text
ISR/DMA → RX 佇列 → 配接器 → 路由器 → 配接器 → TX 佇列 → 驅動/DMA → 線路
```

## 訊息模型與路由

### 標準訊息結構
**訊息標頭設計**
標準訊息格式為所有協定提供通用介面，同時保留特定協定的資訊。

**標頭欄位**
- **來源識別碼**：訊息來源的唯一識別碼
- **目的地識別碼**：訊息目的地的唯一識別碼
- **協定類型**：指示用於傳輸的協定
- **優先權等級**：用於路由和排程的訊息優先權
- **訊息長度**：以位元組為單位的承載資料大小
- **時間戳記**：訊息建立或接收時間
- **序號**：串流中的訊息排序
- **旗標**：額外的控制資訊

**承載資料設計**
- **固定格式**：具有已知配置的結構化資料
- **可變格式**：具有長度指示器的彈性資料
- **二進位格式**：適用於嵌入式系統的高效編碼
- **文字格式**：適用於除錯的人類可讀格式

**訊息驗證**
- **長度檢查**：確保訊息符合協定限制
- **格式驗證**：驗證訊息結構和欄位值
- **校驗和驗證**：偵測傳輸錯誤
- **協定合規性**：確保訊息遵循協定規則

### 路由表設計
**路由表結構**
路由表將訊息特徵對應到輸出介面和轉換。

**路由準則**
- **基於來源的路由**：根據訊息來源進行路由
- **基於目的地的路由**：根據訊息目的地進行路由
- **基於協定的路由**：根據協定類型進行路由
- **基於內容的路由**：根據訊息內容進行路由
- **基於優先權的路由**：根據訊息優先權進行路由

**路由表管理**
- **靜態路由**：在編譯時定義的固定路由規則
- **動態路由**：可在執行時變更的路由規則
- **基於組態的路由**：從組態載入的路由規則
- **基於學習的路由**：從網路行為中學習的路由規則

**路由表最佳化**
- **基於雜湊的查找**：使用雜湊函式進行快速路由表查找
- **基於樹的查找**：適用於階層式路由的高效查找
- **基於快取的查找**：快取常用的路由決策
- **壓縮**：減少路由表記憶體使用量

### 優先權類別與管理
**優先權分類**
- **關鍵優先權**：系統控制和安全訊息
- **高優先權**：即時控制和監控訊息
- **一般優先權**：一般資料和狀態訊息
- **低優先權**：背景和維護訊息

**優先權實作**
- **佇列優先權**：為不同優先權等級設置獨立佇列
- **排程優先權**：較高優先權的任務優先執行
- **資源優先權**：關鍵訊息獲得資源優先
- **丟棄策略**：在負載下丟棄較低優先權的訊息

**優先權繼承**
- **資源繼承**：任務繼承其存取資源的優先權
- **訊息繼承**：訊息繼承其來源的優先權
- **協定繼承**：訊息繼承其協定的優先權
- **動態調整**：根據系統狀態調整優先權

## 流量控制與背壓

### 流量控制基礎
**為什麼流量控制很重要**
多協定系統必須處理不同介面之間變化的資料速率和處理能力。如果沒有適當的流量控制，系統可能會不堪負荷或浪費資源。

**流量控制類型**
- **停止等待**：簡單但效率低的流量控制
- **滑動視窗**：允許多個訊息同時傳輸的高效流量控制
- **基於信用的**：接收方授予發送方信用額度
- **速率限制**：限制資料速率以防止過載

**流量控制實作**
- **硬體流量控制**：使用硬體訊號（RTS/CTS、DTR/DSR）
- **軟體流量控制**：使用軟體協定（XON/XOFF）
- **協定流量控制**：使用特定協定的流量控制機制
- **應用流量控制**：使用應用層級的流量控制

### 背壓策略
**背壓理念**
背壓是系統用來通知無法處理更多資料的機制。有效的背壓可防止系統過載並確保可預測的效能。

**背壓機制**
- **佇列深度限制**：限制佇列中的訊息數量
- **流量控制訊號**：使用協定流量控制機制
- **速率限制**：在系統過載時降低資料速率
- **訊息丟棄**：在負載下丟棄低優先權的訊息

**背壓傳播**
- **即時背壓**：在達到限制時立即發出背壓訊號
- **延遲背壓**：在一段延遲後發出背壓訊號
- **漸進背壓**：隨著負載增加逐漸增加背壓
- **選擇性背壓**：僅對特定來源施加背壓

**背壓策略**
- **尾部丟棄**：當佇列滿時丟棄新訊息
- **優先權丟棄**：優先丟棄低優先權的訊息
- **隨機丟棄**：隨機丟棄訊息以減少同步行為
- **智慧丟棄**：根據內容和重要性丟棄訊息

### 特定協定的流量控制
**UART 流量控制**
- **硬體流量控制**：RTS/CTS 訊號用於即時控制
- **軟體流量控制**：XON/XOFF 字元用於簡單控制
- **基於緩衝區的控制**：監控緩衝區位準並相應地發出訊號
- **基於逾時的控制**：在逾時後發出背壓訊號

**CAN 流量控制**
- **自然仲裁**：CAN 內建的仲裁提供流量控制
- **佇列深度限制**：限制傳輸佇列中的訊息數量
- **速率限制**：控制訊息傳輸速率
- **基於優先權的控制**：使用訊息優先權進行流量控制

**Ethernet 流量控制**
- **IEEE 802.3x 流量控制**：標準 Ethernet 流量控制
- **基於緩衝區的控制**：監控緩衝區位準並發送暫停框架
- **速率限制**：在 MAC 層控制傳輸速率
- **基於 QoS 的控制**：使用服務品質進行流量控制

**SPI/I2C 流量控制**
- **時脈控制**：控制時脈頻率以調節資料速率
- **晶片選擇控制**：使用晶片選擇進行流量控制
- **緩衝區監控**：監控緩衝區位準並相應地發出訊號
- **逾時處理**：優雅地處理通訊逾時

## 排程與延遲

### 任務優先權指派
**優先權指派理念**
任務優先權必須反映不同系統功能的關鍵性和時序需求。

**優先權層級**
- **ISR 優先權**：硬體中斷處理的最高優先權
- **路由優先權**：訊息路由和協定轉譯的高優先權
- **介面優先權**：協定介面處理的中等優先權
- **應用優先權**：業務邏輯的較低優先權
- **背景優先權**：維護任務的最低優先權

**優先權指派準則**
- **時序需求**：即時任務獲得較高優先權
- **關鍵性**：安全關鍵任務獲得較高優先權
- **資源使用**：資源密集型任務可能獲得較低優先權
- **相依性**：具有相依性的任務獲得適當的優先權

**優先權反轉預防**
- **優先權繼承**：任務繼承其存取資源的優先權
- **優先權上限**：資源具有優先權上限以防止反轉
- **資源排序**：以一致的順序存取資源
- **逾時處理**：使用逾時防止無限期阻塞

### 延遲管理
**延遲來源**
- **中斷延遲**：從中斷到 ISR 執行的時間
- **上下文切換延遲**：在任務之間切換的時間
- **佇列延遲**：訊息在佇列中等待的時間
- **處理延遲**：處理訊息的時間
- **傳輸延遲**：傳輸訊息的時間

**延遲預算**
- **端到端延遲**：從來源到目的地的總時間
- **階段延遲**：在每個處理階段花費的時間
- **餘裕分配**：為意外延遲保留時間
- **最壞情況分析**：分析最壞情況的延遲場景

**延遲最佳化**
- **中斷最佳化**：最小化 ISR 執行時間
- **佇列最佳化**：使用高效的佇列實作
- **處理最佳化**：最佳化訊息處理演算法
- **傳輸最佳化**：使用高效的傳輸方法

### DMA 與快取考量
**DMA 使用策略**
- **大量傳輸**：使用 DMA 進行大量資料傳輸
- **中斷減少**：DMA 減少 CPU 中斷負載
- **記憶體效率**：DMA 可以更節省記憶體
- **效能改善**：DMA 改善整體系統效能

**快取管理**
- **快取一致性**：確保 DMA 和 CPU 看到一致的資料
- **緩衝區對齊**：將緩衝區對齊到快取行邊界
- **快取策略**：為 DMA 緩衝區選擇適當的快取策略
- **記憶體屏障**：在需要時使用記憶體屏障

**DMA 緩衝區管理**
- **緩衝區分配**：分配 DMA 安全的緩衝區
- **緩衝區池化**：使用緩衝區池進行高效分配
- **緩衝區生命週期**：管理緩衝區的分配和釋放
- **錯誤處理**：優雅地處理 DMA 錯誤

## 錯誤處理與復原

### 錯誤分類與處理
**錯誤類型**
- **硬體錯誤**：實體層通訊故障
- **協定錯誤**：特定協定的錯誤狀況
- **資料錯誤**：資料損壞或格式違規
- **系統錯誤**：資源耗盡或系統故障

**錯誤處理策略**
- **錯誤偵測**：盡早偵測錯誤
- **錯誤回報**：以足夠的細節回報錯誤
- **錯誤復原**：嘗試從錯誤中復原
- **錯誤記錄**：記錄錯誤以供分析和除錯

**錯誤復原機制**
- **重試機制**：重試失敗的操作
- **後備模式**：切換到替代運作模式
- **錯誤修正**：在可能時修正錯誤
- **系統重置**：在無法復原時重置系統

### 故障隔離與遏制
**故障隔離理念**
系統某部分的故障不應影響其他部分。有效的故障隔離可改善系統可靠性和可維護性。

**隔離機制**
- **程序隔離**：為不同功能使用獨立程序
- **記憶體隔離**：為不同元件使用獨立記憶體空間
- **資源隔離**：為不同功能使用獨立資源
- **介面隔離**：為不同協定使用獨立介面

**遏制策略**
- **錯誤邊界**：定義錯誤被遏制的邊界
- **資源限制**：限制資源使用以防止級聯故障
- **逾時機制**：使用逾時防止無限期阻塞
- **斷路器**：停用故障元件以防止系統故障

**復原機制**
- **自動復原**：從暫時性錯誤中自動復原
- **手動復原**：對於持續性錯誤需要手動介入
- **漸進復原**：逐步恢復系統功能
- **部分復原**：在無法完全復原時恢復部分功能

## 範例路由迴圈實作

### 路由器設計理念
路由器是協調不同協定之間通訊的核心元件。它必須高效、可靠且可維護。

**路由器職責**
- **訊息路由**：將訊息導向適當的目的地
- **協定轉譯**：在不同協定格式之間轉換
- **負載平衡**：將負載分散到多個介面
- **錯誤處理**：處理路由和轉譯錯誤
- **效能監控**：追蹤路由效能指標

**路由器設計原則**
- **單一職責**：路由器僅專注於路由
- **效率**：最小化每個訊息的處理開銷
- **可靠性**：優雅地處理錯誤而不遺失訊息
- **可維護性**：簡單、清晰的程式碼結構
- **可測試性**：易於獨立測試路由邏輯

### 路由器實作
```c
// 範例路由迴圈（虛擬碼 C）
for (;;) {
  message_t *msg = queue_receive(router_q, ROUTER_WAIT_MS);
  if (!msg) continue;

  route_t *r = route_lookup(msg);
  if (!r) { stats.unknown++; buffer_free(msg); continue; }

  for (int i = 0; i < r->num_outputs; i++) {
    adapter_t *ad = r->outputs[i];
    if (!adapter_try_send(ad, msg)) {
      // 套用背壓策略
      if (msg->priority <= PRIORITY_NORMAL) { stats.drop++; }
      else { adapter_blocking_send(ad, msg, TIMEOUT_MS); }
    }
  }

  buffer_free(msg);
}
```

**路由迴圈分析**
- **訊息接收**：從輸入佇列接收訊息
- **路由查找**：為訊息找到適當的路由
- **輸出處理**：將訊息發送到所有相關輸出
- **背壓處理**：在輸出忙碌時套用背壓
- **資源清理**：處理完成後釋放訊息緩衝區

**效能考量**
- **佇列效率**：使用高效的佇列實作
- **路由查找**：最佳化路由查找演算法
- **記憶體管理**：最小化記憶體分配/釋放
- **錯誤處理**：處理錯誤而不影響效能

## 時間同步

### 同步需求
**為什麼同步很重要**
多協定系統經常需要關聯跨不同協定和介面的事件。時間同步使這種關聯成為可能。

**同步類型**
- **時脈同步**：同步系統時脈
- **事件同步**：同步事件時間戳記
- **資料同步**：跨介面同步資料
- **協定同步**：同步協定狀態機

**同步方法**
- **硬體同步**：使用硬體訊號進行同步
- **軟體同步**：使用軟體演算法進行同步
- **網路同步**：使用網路協定進行同步
- **外部同步**：使用外部時間源

### PTP 與網路時間同步
**PTP（精確時間協定）**
- **主從架構**：一個裝置作為時間主機
- **硬體時間戳記**：使用硬體進行精確的時間戳記
- **同步訊息**：定期訊息用於時間同步
- **延遲量測**：量測網路延遲以達到精確同步

**實作考量**
- **硬體支援**：需要硬體時間戳記支援
- **網路需求**：需要網路基礎設施支援
- **組態**：需要仔細的組態以達到最佳效能
- **監控**：監控同步效能

**替代同步方法**
- **NTP（網路時間協定）**：精確度較低但廣泛支援
- **GPS 同步**：使用 GPS 作為絕對時間參考
- **手動同步**：用於簡單系統的手動時間同步
- **無同步**：對於非關鍵應用接受時間差異

## 測試策略與驗證

### 測試理念
**為什麼測試很重要**
多協定系統是複雜的，必須經過徹底測試以確保在所有條件下可靠運作。

**測試目標**
- **功能性**：驗證系統功能正確
- **效能**：驗證系統滿足效能需求
- **可靠性**：驗證系統運作可靠
- **安全性**：驗證系統安全措施

**測試方法**
- **單元測試**：測試個別元件
- **整合測試**：測試元件之間的互動
- **系統測試**：測試完整系統
- **現場測試**：在真實環境中測試

### 測試類型與實作
**浸泡測試**
- **目的**：驗證系統在長時間運行下的穩定性
- **實作**：在負載下長時間運行系統
- **指標**：監控系統效能和錯誤率
- **成功準則**：系統在整個測試期間保持穩定

**故障注入測試**
- **目的**：驗證系統在故障條件下的行為
- **實作**：向系統元件注入故障
- **故障類型**：硬體故障、軟體故障、通訊故障
- **成功準則**：系統優雅地處理故障

**效能測試**
- **目的**：驗證系統滿足效能需求
- **實作**：在各種負載下量測系統效能
- **指標**：延遲、吞吐量、資源使用
- **成功準則**：系統滿足效能規格

**安全測試**
- **目的**：驗證系統安全措施
- **實作**：嘗試破壞系統安全
- **攻擊類型**：各種類型的安全攻擊
- **成功準則**：系統抵禦安全攻擊
