# 即時通訊

即時通訊需要從感測器/致動器到控制器以及整個網路堆疊中具備有界延遲和受控抖動。本指南專注於微控制器和嵌入式 Linux 上的確定性技術。

---

## 🧠 **概念優先**

### **即時 vs 快速**
**概念**：即時系統優先考慮可預測性而非速度。
**重要性**：一個有時非常快但有時很慢的系統，即使平均速度更快，也不是即時系統。
**最小範例**：比較一個平均延遲 1ms 但最壞情況 100ms 的系統，與一個穩定 5ms 延遲的系統。
**試試看**：測量您通訊系統的平均和最壞情況延遲。
**重點**：即時系統必須保證最壞情況效能，而不只是良好的平均效能。

### **延遲預算**
**概念**：您必須為系統中的每個元件分配時間，以滿足整體時序要求。
**重要性**：沒有適當的預算，您無法保證系統能滿足即時限制。
**最小範例**：設計一個包含感測器讀取、處理、通訊和致動的 10ms 控制迴圈。
**試試看**：測量系統中每個元件的延遲並建立預算。
**重點**：每個元件都必須符合其分配的時間預算。

## 核心概念與理論

### 即時通訊基礎
**什麼讓通訊成為「即時」？**
即時通訊不僅僅是速度——而是可預測性。如果系統能保證在指定的時間限制內做出回應，無論系統負載或外部條件如何，該系統就是即時的。

**關鍵即時特性**
- **確定性**：在所有條件下都具有可預測的行為
- **有界延遲**：最大回應時間是已知且有保證的
- **受控抖動**：回應時間的變化是有限且可預測的
- **容錯性**：系統在故障時仍能繼續運作

**即時 vs 高效能**
- **高效能系統**：針對平均情況效能進行最佳化
- **即時系統**：針對最壞情況效能進行最佳化
- **效能取捨**：即時系統可能犧牲峰值效能以換取可預測性
- **設計哲學**：即時系統優先考慮可靠性而非速度

### 延遲與抖動分析
**延遲組成**
了解延遲來源對即時系統設計至關重要：

**端到端延遲分解**
- **感測器處理**：取得和處理感測器資料的時間
- **通訊**：跨網路傳輸資料的時間
- **處理**：分析資料和做出決策的時間
- **致動**：發送命令和執行動作的時間

**抖動來源與分析**
- **時脈抖動**：系統時脈時序的變化
- **中斷抖動**：中斷回應時間的變化
- **排程抖動**：任務排程的變化
- **網路抖動**：網路傳輸時間的變化

**為什麼抖動很重要**
- **控制系統穩定性**：過度的抖動可能使控制迴圈不穩定
- **同步**：高抖動使系統同步變得困難
- **可預測性**：低抖動能實現可預測的系統行為
- **服務品質**：抖動影響感知的系統品質

### 即時系統分類
**硬即時系統**
- **定義**：錯過截止時間會導致系統故障的系統
- **範例**：汽車煞車系統、醫療設備、工業控制
- **要求**：在所有條件下 100% 遵守截止時間
- **設計方法**：保守設計，具有充足的安全裕度

**軟即時系統**
- **定義**：錯過截止時間會降低效能但不會導致故障的系統
- **範例**：多媒體串流、使用者介面回應性
- **要求**：高截止時間遵守率，優雅降級
- **設計方法**：樂觀設計，具有備用機制

**嚴格即時系統**
- **定義**：錯過截止時間會導致資料遺失但不會導致系統故障的系統
- **範例**：資料擷取系統、即時資料庫
- **要求**：高截止時間遵守率，資料完整性保護
- **設計方法**：平衡設計，具有錯誤恢復機制

## 端到端延遲預算

### 延遲預算哲學
**為什麼要做延遲預算？**
延遲預算是將時間分配給不同系統元件的過程，以確保滿足端到端時序要求。沒有適當的預算，系統可能無法滿足即時要求。

**預算分配策略**
- **由上而下方法**：從整體系統要求開始
- **由下而上方法**：從元件能力開始
- **迭代改進**：根據測量結果改進預算
- **安全裕度**：為意外延遲包含裕度

**預算組成**
- **處理時間**：資料處理的 CPU 時間
- **通訊時間**：網路傳輸時間
- **佇列時間**：在佇列中等待的時間
- **排程時間**：任務排程和上下文切換的時間

### 實際延遲預算範例
**系統需求分析**
讓我們考慮一個具有 5ms 週期時間要求的即時控制系統：

**系統概述**
- **應用**：即時馬達控制系統
- **週期時間**：總計 5ms 週期
- **延遲要求**：從感測器到致動器 ≤2ms
- **安全裕度**：總週期時間的 20%

**元件延遲分配**
- **ISR（感測器）+ DMA 完成**：≤100 µs（週期的 2%）
- **複製/解析為訊息**：≤200 µs（週期的 4%）
- **排入即時任務佇列**：≤100 µs（週期的 2%）
- **網路堆疊排入佇列**：≤300 µs（週期的 6%）
- **線路 + 對端處理**：≤1.0 ms（週期的 20%）
- **致動器命令排入佇列**：≤300 µs（週期的 6%）
- **總分配**：2.0 ms（週期的 40%）
- **安全裕度**：1.0 ms（週期的 20%）
- **剩餘裕度**：2.0 ms（週期的 40%）

**預算驗證**
- **測量點**：使用 GPIO 切換來標記階段邊界
- **時序分析**：使用邏輯分析儀擷取時序資料
- **統計分析**：分析最壞情況、平均值和第 99 百分位
- **裕度驗證**：確保實際時序符合預算

## MCU 即時通訊技術

### 中斷與 DMA 最佳化
**中斷設計哲學**
在即時系統中，中斷必須快速且可預測地處理。目標是最小化中斷延遲，同時維持系統回應性。

**中斷最佳化策略**
- **最小 ISR 設計**：盡可能縮短中斷服務常式
- **優先權管理**：使用適當的中斷優先權
- **巢狀控制**：控制中斷巢狀以防止優先權反轉
- **向量表最佳化**：最佳化中斷向量表的放置位置

**DMA 整合**
- **減少中斷**：使用 DMA 減少 CPU 中斷負載
- **緩衝區管理**：預先分配 DMA 緩衝區以獲得可預測的效能
- **快取一致性**：確保 DMA 和 CPU 看到一致的資料
- **錯誤處理**：處理 DMA 錯誤而不影響即時效能

**即時記憶體管理**
- **靜態分配**：預先分配記憶體以避免分配延遲
- **緩衝池**：使用緩衝池進行高效記憶體管理
- **快取最佳化**：最佳化快取使用以獲得即時效能
- **記憶體保護**：使用 MPU/MMU 保護記憶體安全

### 任務優先權與排程
**優先權指派哲學**
任務優先權必須反映不同系統功能的即時需求。較高優先權的任務應處理更具時間關鍵性的操作。

**優先權層次設計**
- **ISR 優先權**：硬體中斷處理的最高優先權
- **即時通訊**：時間關鍵通訊的高優先權
- **控制處理**：控制演算法執行的中優先權
- **背景任務**：非關鍵操作的最低優先權

**優先權繼承與反轉預防**
- **優先權繼承**：任務繼承其存取資源的優先權
- **優先權上限**：資源具有優先權上限以防止反轉
- **資源排序**：以一致的順序存取資源
- **逾時處理**：使用逾時防止無限期阻塞

**排程考量**
- **搶占式排程**：允許較高優先權任務搶占較低優先權任務
- **時間分片**：在相同優先權的任務之間公平分配 CPU 時間
- **截止時間排程**：對時間關鍵任務使用基於截止時間的排程
- **資源排程**：排程資源存取以防止衝突

## 嵌入式 Linux 技術

### 即時核心配置
**即時核心變體**
嵌入式 Linux 提供了多種即時運作選項：

**PREEMPT_RT 修補程式**
- **說明**：Linux 核心的即時搶占修補程式
- **優點**：亞毫秒回應時間，可預測的排程
- **取捨**：增加核心開銷，降低吞吐量
- **使用情境**：硬即時應用，低延遲需求

**低延遲核心**
- **說明**：針對低延遲運作最佳化的核心
- **優點**：在不進行重大核心變更的情況下降低延遲
- **取捨**：有限的即時保證
- **使用情境**：軟即時應用，通用系統

**具最佳化的標準核心**
- **說明**：具有即時最佳化的標準核心
- **優點**：熟悉的環境，良好的效能
- **取捨**：有限的即時保證
- **使用情境**：非關鍵即時應用

**核心配置選項**
- **搶占**：啟用核心搶占以獲得更好的回應性
- **計時器頻率**：增加計時器頻率以獲得更好的解析度
- **中斷處理**：最佳化中斷處理以降低延遲
- **記憶體管理**：配置記憶體管理以適應即時需求

### CPU 隔離與親和性
**CPU 隔離哲學**
CPU 隔離確保即時任務不會被其他系統活動中斷，提供可預測的效能。

**隔離技術**
- **CPU 屏蔽**：保留 CPU 給即時任務使用
- **中斷親和性**：將中斷綁定到特定 CPU
- **行程親和性**：將行程綁定到特定 CPU
- **記憶體親和性**：將記憶體綁定到特定 CPU

**親和性管理**
- **靜態親和性**：固定 CPU 分配以獲得可預測的效能
- **動態親和性**：根據系統負載調整 CPU 分配
- **負載平衡**：在可用 CPU 之間分配負載
- **電源管理**：在親和性決策中考慮功耗

**實作考量**
- **硬體支援**：需要硬體支援 CPU 隔離
- **效能影響**：CPU 隔離可能降低整體系統效能
- **配置複雜度**：CPU 隔離需要仔細配置
- **維護**：CPU 隔離需要持續的維護和監控

### 即時排程
**Linux 即時排程**
Linux 為即時應用提供了多種排程策略：

**SCHED_FIFO（先進先出）**
- **說明**：無時間分片的即時排程
- **優點**：可預測的行為，不會被較低優先權任務搶占
- **取捨**：如果設計不當可能阻塞系統
- **使用情境**：硬即時應用，簡單排程需求

**SCHED_RR（循環排程）**
- **說明**：具有時間分片的即時排程
- **優點**：公平的 CPU 分配，防止任務飢餓
- **取捨**：不如 SCHED_FIFO 可預測
- **使用情境**：軟即時應用，公平排程需求

**SCHED_DEADLINE**
- **說明**：基於截止時間的排程
- **優點**：保證遵守截止時間，高效的資源利用
- **取捨**：配置複雜，工具支援有限
- **使用情境**：複雜即時應用，截止時間需求

**排程配置**
- **優先權指派**：為即時任務指派適當的優先權
- **CPU 親和性**：將任務綁定到特定 CPU 以獲得可預測的效能
- **記憶體鎖定**：鎖定記憶體以防止分頁延遲
- **資源限制**：設定資源限制以防止資源耗盡

## 網路傳輸選擇

### 即時協定選擇
**即時協定要求**
不同的協定為即時通訊提供不同的特性：

**CAN/CAN-FD**
- **即時特性**：自然優先順序，確定性仲裁
- **效能**：最高 1 Mbps（CAN），8 Mbps（CAN-FD）
- **使用情境**：汽車、工業控制、嵌入式系統
- **優點**：內建錯誤偵測，基於優先權的仲裁
- **缺點**：頻寬有限，單主架構

**Ethernet with TSN/AVB**
- **即時特性**：時間感知整形，排程流量
- **效能**：100 Mbps 至 10 Gbps
- **使用情境**：工業自動化，專業音訊/視訊
- **優點**：高頻寬，標準基礎設施
- **缺點**：配置複雜，基礎設施需求

**UDP 用於即時**
- **即時特性**：低開銷，無連線建立
- **效能**：僅受網路容量限制
- **使用情境**：即時串流、遊戲、IoT 應用
- **優點**：實作簡單，低延遲
- **缺點**：無可靠性保證，無流量控制

**TCP 用於即時**
- **即時特性**：可靠傳遞，流量控制
- **效能**：受網路條件和流量控制限制
- **使用情境**：可靠即時通訊，控制系統
- **優點**：內建可靠性，流量控制
- **缺點**：較高延遲，隊頭阻塞

### 即時協定配置
**CAN 配置**
- **位元時序**：配置最佳取樣點和同步
- **訊息優先權**：根據即時需求分配優先權
- **錯誤處理**：根據系統需求配置錯誤處理
- **匯流排利用率**：即時系統的匯流排利用率保持在 70% 以下

**Ethernet TSN 配置**
- **時間同步**：配置 PTP 以實現精確時間同步
- **流量整形**：配置流量整形以獲得可預測的效能
- **排程**：為時間關鍵資料配置排程流量
- **QoS**：配置服務品質以進行優先權處理

**UDP 配置**
- **緩衝區大小**：根據預期流量模式調整緩衝區大小
- **QoS 標記**：使用 DSCP/ToS 進行優先權處理
- **多播**：使用多播進行高效群組通訊
- **錯誤處理**：實作應用層級錯誤處理

**TCP 配置**
- **Nagle 演算法**：對低延遲應用停用 Nagle
- **緩衝區大小**：最佳化緩衝區大小以獲得效能
- **Keepalive**：配置 keepalive 以進行連線監控
- **壅塞控制**：選擇適當的壅塞控制演算法

## 佇列與反壓

### 即時佇列設計
**佇列設計哲學**
即時系統中的佇列必須在所有負載條件下提供可預測的效能。目標是最小化延遲，同時防止緩衝區溢位。

**佇列類型與特性**
- **FIFO 佇列**：實作簡單，行為可預測
- **優先佇列**：處理基於優先權的排程
- **環形緩衝區**：高效的記憶體使用，有界延遲
- **無鎖佇列**：減少競爭，提升效能

**佇列大小策略**
- **流量分析**：分析預期流量模式
- **延遲要求**：根據延遲要求調整佇列大小
- **記憶體限制**：考慮可用記憶體
- **效能要求**：平衡延遲和吞吐量

**佇列管理**
- **水位線管理**：使用水位線進行流量控制
- **溢位處理**：優雅地處理佇列溢位
- **不足處理**：適當地處理佇列不足
- **效能監控**：監控佇列效能指標

### 反壓實作
**反壓哲學**
反壓是系統用來表示無法處理更多資料的機制。在即時系統中，必須快速處理反壓以防止時序違規。

**反壓機制**
- **流量控制信號**：使用協定流量控制機制
- **佇列深度限制**：限制佇列深度以防止溢位
- **速率限制**：在系統過載時降低資料速率
- **訊息丟棄**：在負載下丟棄低優先權訊息

**反壓策略**
- **立即反壓**：一達到限制就發出反壓信號
- **漸進反壓**：隨著負載增加逐漸增加反壓
- **選擇性反壓**：僅對特定來源施加反壓
- **基於優先權的反壓**：根據訊息優先權施加反壓

**反壓處理**
- **來源適應**：來源適應反壓信號
- **負載削減**：在反壓啟動時降低系統負載
- **優雅降級**：在負載下減少功能
- **恢復機制**：在負載降低時恢復功能

## 時間戳記與同步

### 時間同步基礎
**為什麼時間同步很重要**
即時系統通常需要在不同元件和介面之間關聯事件。時間同步能實現這種關聯並改善系統效能。

**同步類型**
- **時脈同步**：同步系統時脈
- **事件同步**：同步事件時間戳記
- **資料同步**：跨介面同步資料
- **協定同步**：同步協定狀態機

**同步方法**
- **硬體同步**：使用硬體信號進行同步
- **軟體同步**：使用軟體演算法進行同步
- **網路同步**：使用網路協定進行同步
- **外部同步**：使用外部時間來源

### PTP 與網路時間同步
**PTP（精確時間協定）**
- **主從架構**：一個裝置作為時間主機
- **硬體時間戳記**：使用硬體產生精確的時間戳記
- **同步訊息**：定期訊息用於時間同步
- **延遲測量**：測量網路延遲以實現精確同步

**PTP 實作考量**

## 實作範例

### 最小 UDP 低延遲路徑
**實作哲學**
目標是建立一條具有最小延遲和抖動的通訊路徑。每項最佳化都必須以效能需求為依據。

**Socket 配置**
```c
// 以 DSCP 設定 socket 以獲得優先權，並調整小緩衝區以降低延遲
int s = socket(AF_INET, SOCK_DGRAM, 0);
int tos = 0x2e; // 加速轉發 DSCP 46（範例）
setsockopt(s, IPPROTO_IP, IP_TOS, &tos, sizeof(tos));
int rxbuf = 8 * 1024, txbuf = 8 * 1024; // 小緩衝區，避免緩衝延遲
setsockopt(s, SOL_SOCKET, SO_RCVBUF, &rxbuf, sizeof(rxbuf));
setsockopt(s, SOL_SOCKET, SO_SNDBUF, &txbuf, sizeof(txbuf));
```

**配置分析**
- **DSCP 標記**：標記封包以進行優先權處理
- **緩衝區大小**：小緩衝區降低延遲但可能增加丟棄
- **Socket 選項**：配置 socket 以獲得最佳效能
- **錯誤處理**：優雅地處理配置錯誤

**關鍵迴圈設計**
```c
// 關鍵迴圈執行最少工作；將繁重處理卸載到另一個執行緒
for (;;) {
  int n = recv(s, buf, sizeof(buf), 0);
  process_minimal(buf, n);
  sendto(s, reply, reply_len, 0, (struct sockaddr*)&peer, sizeof(peer));
}
```

**迴圈最佳化**
- **最少處理**：保持關鍵迴圈盡可能簡單
- **卸載處理**：將繁重處理移至背景執行緒
- **錯誤處理**：處理錯誤而不影響時序
- **效能監控**：監控迴圈效能
