## 🎯 **概述**

控制器區域網路（CAN）是一種穩健的即時通訊協定，專為在嚴苛環境中進行可靠的資料交換而設計。CAN 最初為汽車應用所開發，現已成為需要確定性通訊、容錯能力和即時效能的嵌入式系統標準。

### **核心概念**
- **多主控通訊** — 任何節點在匯流排空閒時皆可傳送資料
- **以訊息為基礎的協定** — 資料以帶有識別碼的框架傳送
- **仲裁機制** — 非破壞性逐位元仲裁以取得匯流排存取權
- **錯誤偵測** — 內建錯誤偵測與故障隔離機制
- **即時效能** — 基於優先權存取的確定性通訊

### **面試官意圖（他們在探查什麼）**
- 你能解釋仲裁機制以及為何 CAN 具有確定性嗎？
- 你瞭解錯誤狀態和故障隔離嗎？
- 你能推論匯流排負載和訊息優先權嗎？

## 🤔 **什麼是 CAN 協定？**

CAN 協定是一種序列通訊標準，使多個電子控制單元（ECU）無需中央電腦即可相互通訊。它採用以訊息為基礎的通訊方式，資料以帶有唯一識別碼的框架傳送，從而在分散式系統中實現高效、可靠且即時的通訊。

### **核心概念**

**多主控架構：**
- **分散式控制**：無中央主控器控制網路
- **點對點通訊**：任何節點皆可發起通訊
- **匯流排存取**：基於優先權仲裁的競爭式存取
- **網路可擴展性**：支援多個節點（通常最多可達 110 個）

**以訊息為基礎的通訊：**
- **框架式傳輸**：資料以結構化框架傳送
- **基於識別碼的路由**：訊息以唯一識別碼識別
- **廣播通訊**：訊息廣播至所有節點
- **過濾機制**：節點根據識別碼過濾訊息

**即時效能：**
- **確定性時序**：可預測的通訊時序
- **基於優先權的存取**：高優先權訊息優先傳送
- **有界延遲**：關鍵訊息保證最大延遲
- **同步操作**：跨網路同步通訊

**容錯能力：**
- **錯誤偵測**：內建錯誤偵測機制
- **故障隔離**：故障節點從網路隔離
- **自動重傳**：失敗訊息自動重傳
- **網路恢復**：錯誤後自動網路恢復

## 🧠 **CAN 協定概念**

### **網路架構**

**CAN 匯流排拓撲：**
- **線性匯流排**：連接所有節點的單一匯流排
- **星狀拓撲**：連接多個節點的中央集線器
- **環狀拓撲**：節點間的環形連接
- **樹狀拓撲**：階層式連接結構

**節點類型：**
- **主控節點**：可發起通訊的節點
- **從屬節點**：回應請求的節點
- **閘道節點**：連接不同網路的節點
- **橋接節點**：連接不同匯流排區段的節點

**匯流排特性：**
- **差動訊號**：CAN_H 與 CAN_L 訊號
- **終端電阻**：匯流排末端的 120Ω 電阻
- **阻抗**：特性阻抗 120Ω
- **長度**：基於鮑率的最大匯流排長度

### **以訊息為基礎的通訊**

**訊息結構：**
- **識別碼欄位**：唯一的訊息識別碼（11 或 29 位元）
- **控制欄位**：訊息類型與資料長度
- **資料欄位**：訊息資料（0-8 位元組）
- **CRC 欄位**：循環冗餘檢查
- **ACK 欄位**：確認欄位

**訊息類型：**
- **資料框架**：在節點間傳送資料
- **遠端框架**：向其他節點請求資料
- **錯誤框架**：指示通訊錯誤
- **超載框架**：指示節點超載狀態

**訊息優先權：**
- **基於識別碼的優先權**：較低的識別碼值具有較高的優先權
- **仲裁過程**：非破壞性逐位元仲裁
- **優先權分配**：應用程式特定的優先權分配
- **優先權管理**：動態優先權管理

### **仲裁與存取控制**

**仲裁過程：**
- **逐位元仲裁**：非破壞性仲裁過程
- **顯性與隱性位元**：位元層級的仲裁機制
- **仲裁時序**：仲裁的時序要求
- **仲裁解析**：同時傳輸的解析

**存取控制：**
- **CSMA/CA**：載波感測多重存取／碰撞避免
- **匯流排閒置偵測**：偵測匯流排閒置狀態
- **傳輸時序**：傳輸的時序要求
- **退避策略**：傳輸失敗的退避策略

**優先權管理：**
- **靜態優先權**：固定優先權分配
- **動態優先權**：動態優先權分配
- **優先權繼承**：優先權繼承機制
- **優先權反轉**：防止優先權反轉

## 🔄 **仲裁機制**

### **仲裁過程**

**逐位元仲裁：**
- **顯性位元**：邏輯 0（顯性狀態）
- **隱性位元**：邏輯 1（隱性狀態）
- **仲裁時序**：仲裁的時序要求
- **仲裁解析**：同時傳輸的解析

**仲裁機制：**
```
節點 A：1011010...（識別碼）
節點 B：1011001...（識別碼）
節點 C：1011000...（識別碼）

仲裁結果：
- 所有節點同時傳送
- 顯性位元（0）覆蓋隱性位元（1）
- 節點 C 贏得仲裁（最低識別碼）
- 節點 A 和 B 停止傳送
- 節點 C 繼續傳送
```

**仲裁時序：**
- **位元時序**：精確的位元時序要求
- **取樣點**：位元偵測的最佳取樣點
- **同步**：跨網路的位元同步
- **抖動容忍度**：時序抖動的容忍度

## 🚀 **CAN-FD 擴充**

### **CAN-FD 概述**

**CAN-FD 特性：**

## 💻 **實作**

### **基本 CAN 組態**

**CAN 控制器組態：**
```c
// CAN 組態結構
typedef struct {
    uint32_t baud_rate;         // 鮑率（位元/秒）
    uint8_t  mode;              // 正常、迴路或靜默模式
    uint8_t  auto_retransmit;   // 自動重傳啟用
    uint8_t  auto_bus_off;      // 自動匯流排關閉恢復
    uint8_t  rx_fifo_locked;    // 接收 FIFO 鎖定模式
    uint8_t  tx_fifo_priority;  // 傳送 FIFO 優先權
} CAN_Config_t;

// 初始化 CAN 控制器
HAL_StatusTypeDef can_init(CAN_HandleTypeDef* hcan, CAN_Config_t* config) {
    hcan->Instance = CAN1;
    hcan->Init.Prescaler = SystemCoreClock / (config->baud_rate * 18);
    hcan->Init.Mode = config->mode;
    hcan->Init.SyncJumpWidth = CAN_SJW_1TQ;
    hcan->Init.TimeSeg1 = CAN_BS1_15TQ;
    hcan->Init.TimeSeg2 = CAN_BS2_2TQ;
    hcan->Init.TimeTriggeredMode = DISABLE;
    hcan->Init.AutoBusOff = config->auto_bus_off;
    hcan->Init.AutoWakeUp = DISABLE;
    hcan->Init.AutoRetransmission = config->auto_retransmit;
    hcan->Init.ReceiveFifoLocked = config->rx_fifo_locked;
    hcan->Init.TransmitFifoPriority = config->tx_fifo_priority;
    
    return HAL_CAN_Init(hcan);
}
```

**訊息傳送：**
```c
// CAN 訊息結構
typedef struct {
    uint32_t id;                // 訊息識別碼
    uint8_t  dlc;               // 資料長度碼
    uint8_t  data[8];           // 訊息資料
    uint8_t  rtr;               // 遠端傳輸請求
    uint8_t  ide;               // 識別碼擴充
} CAN_Message_t;

// 傳送 CAN 訊息
HAL_StatusTypeDef can_transmit(CAN_HandleTypeDef* hcan, CAN_Message_t* message) {
    CAN_TxHeaderTypeDef tx_header;
    
    tx_header.StdId = message->id & 0x7FF;
    tx_header.ExtId = message->id >> 11;
    tx_header.IDE = message->ide;
    tx_header.RTR = message->rtr;
    tx_header.DLC = message->dlc;
    tx_header.TransmitGlobalTime = DISABLE;
    
    uint32_t tx_mailbox;
    return HAL_CAN_AddTxMessage(hcan, &tx_header, message->data, &tx_mailbox);
}
```

### **訊息接收**

**訊息接收：**
```c
// 接收 CAN 訊息
HAL_StatusTypeDef can_receive(CAN_HandleTypeDef* hcan, CAN_Message_t* message) {
    CAN_RxHeaderTypeDef rx_header;
    
    HAL_StatusTypeDef status = HAL_CAN_GetRxMessage(hcan, CAN_RX_FIFO0, &rx_header, message->data);
    if (status == HAL_OK) {
        message->id = rx_header.IDE == CAN_ID_STD ? rx_header.StdId : rx_header.ExtId;
        message->dlc = rx_header.DLC;
        message->rtr = rx_header.RTR;
        message->ide = rx_header.IDE;
    }
    
    return status;
}
```

## 🧪 **引導式實驗**

### **實驗 1：CAN 訊息傳送**
**目標**：瞭解 CAN 訊息結構與傳送。
**設置**：設定 CAN 控制器並連接至 CAN 匯流排。
**步驟**：
1. 以 500 kbps 位元速率初始化 CAN 控制器
2. 設定訊息 ID 與資料長度
3. 傳送一則簡單訊息
4. 在示波器上監控傳輸
5. 驗證訊息確認
**預期結果**：成功傳送 CAN 訊息並獲得正確的確認。

### **實驗 2：CAN 錯誤注入與偵測**
**目標**：測試 CAN 錯誤偵測能力。
**設置**：使用 CAN 分析儀或製造故意的錯誤。
**步驟**：
1. 傳送有效的 CAN 訊息
2. 透過操控訊號引入位元錯誤
3. 觀察錯誤框架產生
4. 測試不同的錯誤類型（位元、填充、格式）
5. 驗證錯誤恢復
**預期結果**：瞭解 CAN 錯誤偵測與恢復機制。

### **實驗 3：CAN 匯流排仲裁**
**目標**：展示 CAN 訊息優先權與仲裁。
**設置**：具有不同訊息優先權的多個 CAN 節點。
**步驟**：
1. 設定具有不同訊息 ID 的節點
2. 開始同時傳輸
3. 監控匯流排的仲裁過程
4. 觀察哪個訊息贏得仲裁
5. 分析優先權與 ID 的關係
**預期結果**：瞭解 CAN 仲裁如何基於訊息 ID 運作。
