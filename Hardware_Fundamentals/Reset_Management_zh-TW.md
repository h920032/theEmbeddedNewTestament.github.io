## 🎯 **概述**

重置管理對嵌入式系統至關重要，確保可靠的啟動、處理系統故障，並維護系統完整性。理解重置機制有助於設計能從各種故障條件中復原的穩健系統。

---

## 🚀 **快速參考：關鍵事實**

- **重置來源**：上電重置、看門狗重置、軟體重置、外部腳位重置、欠壓重置、鎖定重置
- **重置偵測**：檢查 RCC->CSR 暫存器旗標以識別重置原因
- **重置時序**：電源穩定延遲、防彈跳、初始化序列
- **重置復原**：不同重置類型的不同策略
- **重置記錄**：保留重置原因用於診斷和除錯
- **系統初始化**：任何重置事件後的正確啟動序列

## 🧠 **概念基礎**

### **重置管理理念**
重置管理代表嵌入式系統中的一個基本原則：**優雅降級和復原**。重置機制不是讓系統完全失敗，而是提供受控的方式從各種故障條件中復原。此理念實現了：
- **系統可靠性**：從暫態故障和錯誤中復原
- **容錯性**：儘管存在硬體或軟體問題仍能持續運作
- **維護效率**：清楚識別系統問題
- **使用者體驗**：無需手動介入的無縫復原

### **為什麼重置管理很重要**
重置管理至關重要，因為嵌入式系統在不可預測的環境中運作，故障是不可避免的。正確的重置管理實現：
- **可預測的啟動**：任何重置事件後一致的系統行為
- **故障診斷**：清楚識別導致重置的原因
- **系統復原**：針對不同故障類型的適當復原策略
- **資料保護**：跨重置保存關鍵資訊

### **重置設計挑戰**
設計重置管理系統涉及平衡幾個競爭考量：
- **速度 vs 可靠性**：更快的啟動 vs 徹底的初始化
- **簡單性 vs 穩健性**：簡單的重置邏輯 vs 全面的錯誤處理
- **記憶體 vs 效能**：保留狀態 vs 全新開始方法
- **使用者控制 vs 安全性**：手動重置能力 vs 防止意外重置

## 🎯 **核心概念**

### **概念：重置來源偵測與分類**

**為什麼重要**：知道為什麼發生重置對於正確的系統復原至關重要。不同的重置來源需要不同的處理策略，正確的偵測能實現智慧復原決策。

**最小範例**
```c
// 基本重置來源偵測
typedef enum {
    RESET_SOURCE_POR,      // 上電重置
    RESET_SOURCE_WDT,      // 看門狗逾時
    RESET_SOURCE_SOFTWARE, // 軟體啟動
    RESET_SOURCE_EXTERNAL, // 外部腳位
    RESET_SOURCE_UNKNOWN   // 未知原因
} reset_source_t;

// 從硬體旗標偵測重置來源
reset_source_t detect_reset_source(void) {
    uint32_t reset_flags = RCC->CSR;
    
    if (reset_flags & RCC_CSR_PORRSTF) {
        return RESET_SOURCE_POR;
    } else if (reset_flags & RCC_CSR_WWDGRSTF) {
        return RESET_SOURCE_WDT;
    } else if (reset_flags & RCC_CSR_SFTRSTF) {
        return RESET_SOURCE_SOFTWARE;
    } else if (reset_flags & RCC_CSR_PINRSTF) {
        return RESET_SOURCE_EXTERNAL;
    }
    
    return RESET_SOURCE_UNKNOWN;
}
```

**試試看**：為你的特定微控制器實作重置來源偵測，並用不同的重置情境測試。

**重點摘要**
- 始終在系統初始化的早期檢查重置旗標
- 不同的重置來源需要不同的復原策略
- 記錄重置來源用於除錯和診斷
- 偵測後清除重置旗標

### **概念：重置時序與電源穩定**

**為什麼重要**：正確的重置時序確保可靠的系統啟動。電源穩定、時脈穩定和周邊初始化都需要特定的時序考量以防止啟動失敗。

**最小範例**
```c
// 基本重置時序設定
typedef struct {
    uint32_t power_stabilization_ms;  // 電源穩定時間
    uint32_t clock_settling_ms;       // 時脈振盪器穩定時間
    uint32_t peripheral_init_ms;      // 周邊初始化時間
    uint32_t total_startup_ms;        // 總啟動時間
} reset_timing_t;

// 設定重置時序延遲
void configure_reset_timing(reset_timing_t *timing) {
    // 設定電源穩定延遲
    timing->power_stabilization_ms = 100;  // 100ms 讓電源穩定
    
    // 設定時脈穩定時間
    timing->clock_settling_ms = 50;        // 50ms 讓振盪器穩定
    
    // 計算總啟動時間
    timing->total_startup_ms = timing->power_stabilization_ms + 
                               timing->clock_settling_ms + 
                               timing->peripheral_init_ms;
}
```

**試試看**：量測你的系統實際上電時間，並據此調整時序參數。

**重點摘要**
- 電源需要時間穩定後系統才能啟動
- 時脈振盪器需要穩定時間才能穩定運作
- 不同周邊可能需要不同的初始化時序
- 在各種電源條件下測試時序

### **概念：重置復原策略與狀態管理**

**為什麼重要**：不同的重置情境需要不同的復原方法。了解什麼狀態可以保留、什麼必須重新初始化，能實現高效復原並維護系統完整性。

**最小範例**
```c
// 重置復原策略選擇
typedef enum {
    RECOVERY_COLD_START,   // 完整系統重新初始化
    RECOVERY_WARM_START,   // 部分重新初始化
    RECOVERY_HOT_START     // 最小重新初始化
} recovery_strategy_t;

// 根據重置來源選擇復原策略
recovery_strategy_t select_recovery_strategy(reset_source_t source) {
    switch (source) {
        case RESET_SOURCE_POR:
            return RECOVERY_COLD_START;  // 需要完整初始化
            
        case RESET_SOURCE_SOFTWARE:
            return RECOVERY_WARM_START;  // 部分初始化
            
        case RESET_SOURCE_WDT:
            return RECOVERY_HOT_START;   // 最小初始化
            
        default:
            return RECOVERY_COLD_START;  // 預設為安全選項
    }
}
```

**試試看**：實作不同的復原策略，並測試各種重置類型後的系統行為。

**重點摘要**
- 上電重置需要完整的系統初始化
- 軟體重置可以保留部分系統狀態
- 看門狗重置可能只需要最小的復原
- 復原後始終驗證系統狀態

## 🧪 **引導實驗**

### **實驗 1：重置來源偵測與記錄**
**目標**：實作一個偵測和記錄不同重置來源的系統。

**步驟**：
1. 使用硬體旗標設定重置來源偵測
2. 實作重置記錄到非揮發性記憶體
3. 測試不同的重置情境（電源循環、看門狗、軟體）
4. 驗證重置來源識別的準確性

**預期結果**：理解重置偵測機制和正確的旗標處理。

### **實驗 2：重置時序與上電序列**
**目標**：量測和最佳化系統啟動時序。

**步驟**：
1. 使用示波器量測實際上電時間
2. 實作可設定的啟動延遲
3. 在各種電源條件下測試啟動
4. 最佳化時序以實現可靠運作

**預期結果**：獲得重置時序和電源考量的實務經驗。

### **實驗 3：重置復原策略實作**
**目標**：為各種重置類型實作不同的復原策略。

**步驟**：
1. 實作冷啟動、暖啟動和熱啟動復原策略
2. 測試不同重置來源後的復原行為
3. 復原後驗證系統狀態
4. 量測每種策略的復原時間

**預期結果**：理解重置復原機制和狀態管理。
