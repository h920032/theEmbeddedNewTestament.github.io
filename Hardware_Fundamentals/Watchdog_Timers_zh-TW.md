## 🎯 **概述**

看門狗定時器是監控系統健康的重要安全機制，當系統變得無回應或進入錯誤狀態時自動重置系統。它們對可靠的嵌入式系統至關重要，特別是在安全關鍵應用中。

---

## 🚀 **快速參考：關鍵事實**

- **系統監控**：持續的健康檢查和故障偵測
- **逾時週期**：看門狗餵食之間的最大時間
- **復原機制**：自動系統重置和復原
- **故障偵測**：識別系統故障和錯誤
- **硬體 vs 軟體**：獨立硬體定時器 vs 基於軟體的監控
- **視窗模式**：餵食的時間窗口限制以捕捉「太快」的故障
- **復原等級**：根據故障嚴重程度的不同復原策略

## 🔍 **視覺化理解**

### **看門狗定時器運作流程**
```
系統運行 → 健康檢查 → 餵食看門狗 → 繼續運作
     ↓              ↓              ↓              ↓
   正常        監控任務      重置定時器    系統正常
   運作        檢查記憶體    延長存活      繼續

     ↓              ↓              ↓              ↓
   故障         健康檢查     未餵食       逾時
   偵測         失敗          看門狗       重置
```

### **看門狗類型比較**
```
硬體看門狗：    CPU 時脈 → 獨立定時器 → 重置電路
                            ↓              ↓              ↓
                      獨立於        即使 CPU         可靠的重置
                      CPU 狀態      崩潰仍運行       機制

軟體看門狗：    系統任務 → 監控其他 → 觸發復原
                            ↓              ↓              ↓
                      屬於作業系統  可能隨系統       基於軟體的
                      軟體一部分    一起失敗         復原
```

### **復原策略層次**
```
故障偵測 → 評估嚴重程度 → 選擇復原 → 執行復原
     ↓              ↓              ↓              ↓
   系統錯誤     輕微故障     重啟任務      繼續
   重大故障     重大故障     重啟應用程式  運作
   嚴重故障     嚴重故障     系統重置      完整復原
```

## 🧠 **概念基礎**

### **看門狗作為安全網**
看門狗定時器代表嵌入式系統中的一個基本安全原則：**故障安全運作**。看門狗定時器不是讓系統靜默地失敗或變得無回應，而是提供自動偵測和復原機制。此理念實現了：
- **系統可靠性**：從暫態故障自動復原
- **容錯性**：儘管存在軟體或硬體問題仍能持續運作
- **安全保證**：保證系統對關鍵故障的回應
- **維護效率**：清楚識別系統問題

### **為什麼看門狗定時器很重要**
看門狗定時器至關重要，因為嵌入式系統在不可預測的環境中運作，故障是不可避免的。正確的看門狗實作實現：
- **自動復原**：無需手動介入的系統自我修復
- **故障偵測**：早期識別系統問題
- **系統完整性**：防止系統鎖定或掛起狀況
- **安全合規**：滿足關鍵應用的安全標準

### **看門狗設計挑戰**
設計看門狗系統涉及平衡幾個競爭考量：
- **逾時 vs 效能**：更短的逾時提供更快的復原但需要更頻繁的餵食
- **簡單性 vs 穩健性**：簡單的看門狗 vs 全面的健康監控
- **硬體 vs 軟體**：獨立硬體 vs 基於軟體的監控
- **復原 vs 延續性**：立即重置 vs 優雅降級

## 🎯 **核心概念**

### **概念：硬體看門狗設定與操作**

**為什麼重要**：硬體看門狗提供最可靠的系統監控，因為它們獨立於主 CPU 運作。即使 CPU 完全無回應，它們也能偵測和復原系統故障。

**最小範例**
```c
// 基本硬體看門狗設定
typedef struct {
    uint32_t timeout_ms;          // 逾時週期（毫秒）
    uint32_t prescaler;           // 時脈預除頻值
    bool window_mode;             // 啟用視窗模式
} hw_watchdog_config_t;

// 初始化硬體看門狗
void init_hardware_watchdog(hw_watchdog_config_t *config) {
    // 啟用看門狗時脈（LSI = 40kHz）
    RCC->CSR |= RCC_CSR_LSION;
    
    // 等待 LSI 就緒
    while (!(RCC->CSR & RCC_CSR_LSIRDY));
    
    // 設定預除頻器和重載值
    IWDG->PR = config->prescaler;
    IWDG->RLR = (config->timeout_ms * 40) / (1000 * (config->prescaler + 1));
    
    // 啟用並啟動看門狗
    IWDG->KR = 0xCCCC;  // 啟用
    IWDG->KR = 0xAAAA;  // 啟動
}
```

**試試看**：設定一個 1 秒逾時的硬體看門狗並測試系統復原。

**重點摘要**
- 硬體看門狗使用獨立的時脈來源（LSI）
- 預除頻器和重載值決定逾時週期
- 看門狗必須在逾時前被餵食以防止重置
- 獨立運作確保即使 CPU 故障也能可靠運作

### **概念：系統健康監控和看門狗餵食策略**

**為什麼重要**：有效的看門狗運作需要智慧的健康監控。僅靠定時器餵食看門狗並不能保證系統健康 — 餵食應該只在關鍵系統功能被驗證為正常運作時才進行。

**最小範例**
```c
// 系統健康監控結構
typedef struct {
    bool tasks_running;           // 關鍵任務正在運行
    bool memory_ok;              // 記憶體完整性檢查通過
    bool communication_ok;        // 通訊系統運作中
    bool sensors_responding;      // 感測器資料有效
} system_health_t;

// 僅在系統健康時餵食看門狗
void feed_watchdog_if_healthy(void) {
    system_health_t health = check_system_health();
    
    if (health.tasks_running && 
        health.memory_ok && 
        health.communication_ok && 
        health.sensors_responding) {
        
        // 系統健康，餵食看門狗
        IWDG->KR = 0xAAAA;
    } else {
        // 系統有問題，讓看門狗重置
        // 記錄健康狀態用於除錯
        log_system_health(&health);
    }
}
```

**試試看**：實作一個在餵食看門狗前檢查多個系統元件的健康監控系統。

**重點摘要**
- 健康檢查應驗證關鍵系統功能
- 僅在系統真正健康時餵食看門狗
- 記錄健康狀態用於除錯和分析
- 如果系統不健康就讓看門狗重置

### **概念：復原策略選擇與實作**

**為什麼重要**：不同類型的系統故障需要不同的復原方法。實作多個復原等級能實現優雅降級，並防止對輕微問題進行不必要的系統重置。

**最小範例**
```c
// 復原策略等級
typedef enum {
    RECOVERY_NONE,           // 不需要復原
    RECOVERY_RESTART_TASK,   // 重啟失敗的任務
    RECOVERY_RESTART_APP,    // 重啟應用程式
    RECOVERY_SYSTEM_RESET    // 完整系統重置
} recovery_level_t;

// 根據故障類型決定復原策略
recovery_level_t determine_recovery_strategy(system_health_t *health) {
    if (!health->tasks_running) {
        return RECOVERY_RESTART_TASK;
    } else if (!health->memory_ok) {
        return RECOVERY_RESTART_APP;
    } else if (!health->communication_ok && !health->sensors_responding) {
        return RECOVERY_SYSTEM_RESET;
    }
    
    return RECOVERY_NONE;
}

// 執行復原策略
void execute_recovery(recovery_level_t strategy) {
    switch (strategy) {
        case RECOVERY_RESTART_TASK:
            restart_failed_tasks();
            break;
        case RECOVERY_RESTART_APP:
            restart_application();
            break;
        case RECOVERY_SYSTEM_RESET:
            system_reset();
            break;
        default:
            break;
    }
}
```

**試試看**：實作一個適當處理不同類型故障的多級復原系統。

**重點摘要**
- 不同故障需要不同的復原策略
- 在系統重置前先實作優雅降級
- 記錄復原動作用於系統分析
- 徹底測試復原機制
