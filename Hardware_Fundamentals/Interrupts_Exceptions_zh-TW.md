## 🎯 **概述**

中斷和例外是允許嵌入式系統高效地回應外部事件和處理錯誤的基本機制。了解中斷處理對即時系統設計和可靠的嵌入式應用至關重要。

---

## 🚀 **快速參考：重點摘要**

- **中斷向量表**：將中斷源映射到處理器函式
- **中斷服務常式（ISR）**：中斷發生時執行的函式
- **中斷延遲**：從中斷發生到處理器開始執行的時間
- **例外處理**：管理系統錯誤和故障
- **優先級層級**：ARM Cortex-M 支援多達 256 個優先級層級
- **巢狀**：較高優先級中斷可搶占較低優先級中斷
- **上下文儲存**：CPU 在 ISR 執行期間自動儲存/還原暫存器

- 你能否設計短小、安全且確定性的 ISR？
- 你是否了解延遲、優先級和巢狀的權衡？
- 你能否解釋常見的故障/例外根本原因？

## 🔍 **視覺化理解**

### **中斷處理流程**
```
正常執行 → 中斷發生 → 上下文儲存 → ISR 執行 → 上下文還原 → 恢復執行
     ↓                    ↓              ↓           ↓              ↓              ↓
   主迴圈          硬體事件      儲存暫存器   處理 IRQ   還原暫存器     繼續
```

### **中斷優先級與巢狀**
```
優先級 0（最高）──┐
                  ├── 可中斷任何較低優先級
優先級 1 ─────────┤
                  ├── 無法中斷優先級 0
優先級 2 ─────────┤
                  └── 無法中斷優先級 0 或 1
優先級 3（最低）──┘
```

### **例外 vs 中斷處理**
```
例外（內部）            中斷（外部）
     ↓                          ↓
硬體故障              周邊事件
記憶體違規            GPIO 變化
非法指令              計時器到期
堆疊溢位              通訊事件
```

## 🧠 **概念基礎**

### **中斷驅動範式**
中斷代表從輪詢式系統到事件驅動架構的根本轉變。系統不再持續檢查事件，而是等待硬體發出需要處理的訊號。這個範式啟用：
- **高效資源使用**：CPU 在需要前進入睡眠
- **即時回應**：對關鍵事件的立即回應
- **功耗最佳化**：電池供電系統減少活躍時間

### **為什麼中斷和例外重要**
中斷是即時嵌入式系統的骨幹。它們啟用：
- **回應式使用者介面**：對按鈕按下的立即回應
- **高效通訊**：僅在資料可用時處理資料
- **系統可靠性**：優雅地處理錯誤和故障
- **資源管理**：協調多個硬體周邊設備

### **中斷設計挑戰**
設計中斷系統涉及平衡多個相互競爭的關注點：
- **延遲 vs 複雜度**：更快的回應需要更簡單的 ISR
- **優先級 vs 公平性**：關鍵中斷需要立即處理
- **安全性 vs 效能**：穩健的錯誤處理 vs 最小開銷

## 🎯 **核心概念**

### **概念：中斷向量表與處理器註冊**

**重要性**：向量表是中斷處理的中樞神經系統。它將每個中斷源映射到其對應的處理器函式，使 CPU 能在中斷發生時跳轉到正確的程式碼。

**最小範例**
```c
// 基本中斷處理器註冊
typedef void (*interrupt_handler_t)(void);

// 簡單向量表結構
typedef struct {
    interrupt_handler_t reset_handler;
    interrupt_handler_t nmi_handler;
    interrupt_handler_t hardfault_handler;
    interrupt_handler_t irq_handlers[16];  // 外部中斷
} vector_table_t;

// 為特定中斷註冊處理器
void register_interrupt_handler(uint8_t irq_number, interrupt_handler_t handler) {
    if (irq_number < 16) {
        vector_table.irq_handlers[irq_number] = handler;
    }
}
```

**試試看**：建立一個包含計時器和 UART 中斷處理器的簡單向量表。

**重點摘要**
- 向量表必須正確對齊在記憶體中
- 處理器函式必須具有正確的呼叫慣例
- 未使用的中斷應指向預設處理器

### **概念：中斷服務常式設計原則**

**重要性**：ISR 設計直接影響系統回應性和可靠性。設計不當的 ISR 可能導致錯過中斷、優先級反轉和系統不穩定。

**最小範例**
```c
// 良好的 ISR 設計 - 最小且快速
volatile bool data_ready = false;
volatile uint8_t received_data = 0;

void uart_rx_isr(void) {
    // 立即清除中斷旗標
    UART1->SR &= ~UART_SR_RXNE;
    
    // 最小化處理 - 僅捕獲資料
    received_data = UART1->DR;
    data_ready = true;
    
    // 讓主迴圈處理資料
}
```

**試試看**：為 GPIO 按鈕按下設計一個設定旗標供主迴圈處理的 ISR。

**重點摘要**
- 保持 ISR 短小且確定
- 避免函式呼叫和複雜操作
- 使用旗標與主迴圈通訊
- 儘早清除中斷旗標

### **概念：中斷優先級與巢狀管理**

**重要性**：適當的優先級管理確保關鍵中斷能立即獲得處理，同時防止優先級反轉並確保系統回應性。

**最小範例**
```c
// 配置中斷優先級
void configure_interrupt_priorities(void) {
    // 系統計時器設為最高優先級（最低數字）
    NVIC_SetPriority(SysTick_IRQn, 0);
    
    // UART 設為中優先級
    NVIC_SetPriority(UART1_IRQn, 2);
    
    // GPIO 設為較低優先級
    NVIC_SetPriority(EXTI0_IRQn, 4);
    
    // 啟用所有中斷
    NVIC_EnableIRQ(SysTick_IRQn);
    NVIC_EnableIRQ(UART1_IRQn);
    NVIC_EnableIRQ(EXTI0_IRQn);
}
```

**試試看**：配置三個具有不同優先級的中斷並觀察巢狀行為。

**重點摘要**
- 較低的優先級數字 = 較高的優先級
- 較高優先級中斷可搶占較低優先級中斷
- 考慮系統範圍的優先級策略
- 徹底測試優先級互動

### **概念：例外處理與故障恢復**

**重要性**：例外代表必須優雅處理的系統錯誤。適當的例外處理可防止系統當機並啟用瞬時故障的恢復。

**最小範例**
```c
// 基本硬體故障處理器
void hardfault_handler(void) {
    // 捕獲故障資訊
    uint32_t fault_address = SCB->MMFAR;  // 記憶體故障位址
    uint32_t fault_status = SCB->CFSR;    // 組合故障狀態
    
    // 記錄故障資訊（如果可能）
    log_fault_info(fault_address, fault_status);
    
    // 嘗試恢復或重設
    if (can_recover_from_fault(fault_status)) {
        // 嘗試繼續
        return;
    } else {
        // 如果無法恢復則重設系統
        system_reset();
    }
}
```

**試試看**：實作一個記錄錯誤資訊並嘗試恢復的故障處理器。

**重點摘要**
- 始終為關鍵故障設置例外處理器
- 記錄故障資訊以供除錯
- 盡可能實作恢復機制
- 如果恢復失敗則重設系統

## 🧪 **引導式實驗**

### **實驗 1：中斷延遲量測**
**目標**：量測從中斷觸發到 ISR 執行的時間。

**步驟**：
1. 配置已知頻率的計時器中斷
2. 使用 GPIO 切換標記中斷進入/退出
3. 使用示波器或邏輯分析儀量測時序
4. 比較量測值與理論延遲

**預期結果**：了解中斷開銷和影響延遲的因素。

### **實驗 2：優先級巢狀展示**
**目標**：觀察中斷優先級如何影響巢狀行為。

**步驟**：
1. 配置具有不同優先級的多個中斷
2. 同時觸發中斷
3. 觀察執行順序和巢狀
4. 分析優先級反轉場景

**預期結果**：了解中斷優先級機制和巢狀行為。

### **實驗 3：例外處理實作**
**目標**：實作穩健的系統故障例外處理。

**步驟**：
1. 設定常見故障的例外處理器
2. 實作故障記錄和恢復機制
3. 測試故障場景（無效記憶體存取、除以零）
4. 驗證恢復行為

**預期結果**：獲得例外處理和故障恢復的實務經驗。
